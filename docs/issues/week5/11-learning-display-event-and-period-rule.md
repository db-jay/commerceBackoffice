## 목적
- 주니어가 `display/event`를 단순 조회 테이블이 아니라 "노출 정책과 기간 규칙을 가진 운영 컨텍스트"로 설명할 수 있게 만든다.

## 숙지해야 할 내용
- 왜 `catalog` 상품 정보와 `display` 노출 정책을 같은 컨텍스트로 두지 않는가
- 이벤트 활성 여부를 판단할 때 상태값과 기간 조건을 함께 봐야 하는 이유는 무엇인가
- `startAt`, `endAt` 같은 시간 규칙에서 경계값 테스트가 왜 중요한가
- 이벤트 적용 여부를 Controller가 아니라 Application/Domain에서 판단해야 하는 이유는 무엇인가
- 대상 상품 미일치, 비활성 이벤트, 기간 외 이벤트가 각각 어떤 운영 의미를 가지는가

## 스스로 설명해보기
- "상품은 존재하지만 지금 전시되면 안 되는 상태"를 어떤 규칙으로 표현할 수 있는가?
- `catalog`와 `display`를 분리하면 어떤 변경이 쉬워지고, 어떤 복잡성이 생기는가?
- 시간 기반 로직을 `LocalDateTime.now()`에 직접 묶으면 테스트에서 어떤 문제가 생기는가?
- 이벤트 기간 판단을 SQL 한 줄로만 끝내지 않고 도메인 규칙으로 드러내야 하는 이유는 무엇인가?
